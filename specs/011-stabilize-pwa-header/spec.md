## User Scenarios & Testing *(mandatory)*

### User Story 1 - Reach Top Header Actions Reliably (Priority: P1)

휴대폰에서 설치형 PWA로 Hinear를 사용하는 사용자는 화면 맨 위의 주요 헤더 액션을 항상 눌러서 사용할 수 있기를 원한다. 상단 제어영역과 앱 헤더가 겹치면 알림, 뒤로가기, 주요 이동 같은 상단 액션이 막혀 핵심 흐름이 바로 중단된다.

**Why this priority**: 상단 액션이 눌리지 않으면 사용자는 핵심 화면으로 진입한 직후부터 흐름이 막히므로 가장 먼저 해결해야 한다.

**Independent Test**: 휴대폰 PWA 환경에서 주요 화면을 열었을 때 상단 헤더의 터치 가능한 항목을 첫 시도에 누를 수 있고, 가려지거나 반응하지 않는 영역이 없으면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 설치형 PWA를 휴대폰에서 열어 주요 화면에 진입한 상태, **When** 화면 상단의 헤더 액션을 누르면, **Then** 각 액션은 휴대폰 제어영역과 겹치지 않은 상태로 즉시 반응해야 한다.
2. **Given** 기기마다 상단 여백 크기가 다른 상태, **When** 사용자가 같은 주요 화면을 열면, **Then** 헤더 내용은 잘리거나 가려지지 않고 터치 가능한 상태로 유지되어야 한다.

---

### User Story 2 - Use Notification Entry Points Without Blocked Touch Targets (Priority: P2)

사용자는 알림과 관련된 상단 진입점이나 설정 진입점을 눌렀을 때 정확히 반응하기를 원한다. 알림 관련 컨트롤이 상단 겹침이나 비정상 배치 때문에 눌리지 않으면 중요한 업데이트 확인과 알림 활성화 흐름이 끊긴다.

**Why this priority**: 상단 헤더 문제 중에서도 알림은 사용자가 상태 변화를 놓치게 만드는 영향이 커서 두 번째 우선순위가 필요하다.

**Independent Test**: 사용자가 휴대폰 PWA에서 알림 관련 버튼이나 진입점을 눌렀을 때 첫 시도에 정상 동작하고, 권한 상태에 맞는 다음 화면이나 안내가 열리면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 휴대폰 PWA에서 알림 관련 상단 액션 또는 설정 진입점을 본 상태, **When** 해당 컨트롤을 누르면, **Then** 다른 화면 요소나 기기 제어영역에 가로막히지 않고 의도한 결과가 발생해야 한다.
2. **Given** 사용자의 알림 권한 상태가 아직 결정되지 않았거나 이미 결정된 상태, **When** 사용자가 알림 관련 진입점을 누르면, **Then** 시스템은 현재 상태에 맞는 다음 단계나 설명을 일관되게 보여줘야 한다.

### User Story 3 - Keep the Top Layout Stable Across Mobile PWA Contexts (Priority: P3)

사용자는 화면을 다시 열거나 기기를 회전하거나 다른 모바일 환경에서 접속해도 상단 레이아웃이 갑자기 밀리거나 겹치지 않기를 원한다. 한 번만 맞고 다른 상황에서 깨지면 PWA 품질에 대한 신뢰가 떨어진다.

**Why this priority**: 핵심 문제를 고친 뒤에도 기기별 편차로 다시 깨지면 동일한 불만이 반복되므로 안정성 기준이 필요하다.

**Independent Test**: 같은 계정으로 여러 모바일 PWA 상황에서 주요 화면을 열었을 때 상단 레이아웃이 일관되고, 터치 가능 영역이 유지되면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 휴대폰 PWA를 다시 실행하거나 백그라운드에서 복귀한 상태, **When** 주요 화면이 다시 표시되면, **Then** 상단 헤더는 이전과 같은 안전한 위치와 터치 영역을 유지해야 한다.
2. **Given** 사용자가 세로와 가로 방향 또는 서로 다른 모바일 기기에서 앱을 사용하는 상태, **When** 주요 화면을 열면, **Then** 상단 레이아웃은 겹침 없이 재배치되어 같은 기능 접근성을 유지해야 한다.

### Edge Cases

- 상단 노치, 다이내믹 아일랜드, 상태 표시 영역 크기가 큰 기기에서도 헤더 액션이 잘리거나 가려지지 않아야 한다.
- 설치형 PWA가 아닌 일반 모바일 브라우저로 열어도 상단 컨트롤 접근성이 악화되지 않아야 한다.
- 알림 권한이 이미 거부되었거나 기기에서 알림을 지원하지 않는 경우에도 상단 진입점은 눌릴 수 있어야 하며 현재 제한 상태를 이해할 수 있어야 한다.
- 앱이 복귀하면서 브라우저 UI 높이가 바뀌거나 화면 회전이 발생해도 상단 헤더의 터치 가능 영역이 사라지지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 모바일 휴대폰 환경에서 주요 화면의 최상단 인터페이스가 기기 제어영역과 물리적으로 겹치지 않도록 표시해야 한다.
- **FR-002**: 시스템은 상단 헤더에 노출된 주요 액션이 시각적으로 보이는 위치와 실제 터치 가능한 위치가 일치하도록 유지해야 한다.
- **FR-003**: 사용자는 모바일 PWA 환경에서 상단 헤더의 주요 액션을 첫 시도에 실행할 수 있어야 한다.
- **FR-004**: 시스템은 현재 모바일 범위에 포함된 알림 관련 진입점, 특히 프로필/설정 화면의 알림 컨트롤이 다른 요소에 가려지거나 눌리지 않는 상태를 남기지 않아야 한다.
- **FR-005**: 시스템은 알림 관련 진입점이 현재 권한 상태와 관계없이 접근 가능해야 하며, 사용자가 다음에 취할 수 있는 행동 또는 현재 제한 상태를 이해할 수 있게 해야 한다.
- **FR-006**: 시스템은 앱 재실행, 화면 복귀, 화면 회전, 기기별 상단 영역 차이 이후에도 상단 레이아웃과 터치 가능 영역을 안정적으로 유지해야 한다.
- **FR-007**: 시스템은 상단 헤더가 안전한 위치로 이동하더라도 화면 제목, 주요 이동, 알림 관련 진입점의 의미와 사용 흐름을 유지해야 한다.
- **FR-008**: 시스템은 이번 범위에 포함된 모바일 주요 화면에 대해 상단 헤더 접근성 검증 기준을 문서화할 수 있어야 한다.

### Key Entities *(include if feature involves data)*

- **Top Safe Zone**: 모바일 기기 상단에서 앱 콘텐츠가 겹치지 않아야 하는 보호 영역을 나타낸다.
- **Header Action Target**: 사용자가 상단 헤더에서 누르는 알림, 이동, 닫기, 되돌아가기 등의 개별 인터랙션 대상을 나타낸다.
- **Notification Entry Point**: 사용자가 알림 상태를 확인하거나 알림 관련 흐름으로 진입하는 상단 또는 설정 영역의 컨트롤을 나타낸다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 범위에 포함된 모바일 주요 화면에서 상단 헤더 컨트롤이 기기 제어영역과 겹쳐 눌리지 않는 사례가 0건이어야 한다.
- **SC-002**: 사용자는 휴대폰 PWA 환경에서 범위에 포함된 상단 헤더 액션을 첫 시도에서 95% 이상 성공적으로 실행할 수 있어야 한다.
- **SC-003**: 사용자는 알림 관련 진입점을 눌렀을 때 3초 이내에 올바른 다음 상태, 화면, 또는 제한 안내를 확인할 수 있어야 한다.
- **SC-004**: 세로/가로 전환, 앱 재실행, 화면 복귀를 포함한 모바일 검증 흐름에서 상단 레이아웃 깨짐으로 인해 핵심 작업이 중단되는 단계가 없어야 한다.

## Assumptions

- 이번 작업의 우선 범위는 사용 빈도가 높은 모바일 주요 화면과 그 상단 헤더이며, 데스크톱 전용 레이아웃 재설계는 포함하지 않는다.
- 사용자 불편의 핵심 원인은 상단 안전영역 처리와 터치 가능 영역 불일치이며, 정보 구조 자체를 새로 바꾸는 것은 이번 범위 밖이다.
- 알림 기능 자체를 새로 확장하는 것보다, 기존 알림 진입과 권한 흐름이 모바일 PWA에서 막히지 않도록 안정화하는 것이 이번 작업의 목적이다.
- 현재 1차 범위의 알림 진입점은 모바일 프로필/설정 화면의 알림 컨트롤이며, 별도의 모바일 상단 헤더 알림 아이콘 추가나 신규 알림 네비게이션 설계는 포함하지 않는다.
- 주요 화면별 상단 헤더 접근성 검증 기준을 남기면 이후 유사 회귀를 줄이는 데 충분한 운영 기준이 된다.
